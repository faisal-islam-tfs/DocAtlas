DocAtlas Pipeline Overview
==========================

1) Discover & validate
- Scan input folder for supported files (.pdf, .doc, .docx, .ppt, .pptx, .xlsx)
- Auto-unpack any .zip archives into a temporary staging area and include supported files found inside them
- Store workbook-facing source paths relative to the selected input root
- Skip unsupported types
- CLI can run with parallel workers (`--workers N`); GUI runs the progress-callback path

2) Text extraction
- .docx -> python-docx
- .doc -> convert to .docx via LibreOffice (soffice), then python-docx
- .pptx -> python-pptx
- .xlsx -> openpyxl
- .pdf -> pdfplumber
  - If text is too short, OCR runs:
    - OCRmyPDF (non-forced pass)
    - OCRmyPDF (forced pass only if still low-text)
    - If needed, Tesseract fallback
  - OCR is used only for PDFs with low/empty extracted text
  - OCR does not use the LLM; it is a separate local process

3) PDF article splitting
- Disabled by default
- Enable with `--articles` (CLI) or GUI article toggle
- When enabled, PDFs are split into articles using conservative strong-heading heuristics
- If structural evidence is weak/ambiguous, the PDF is treated as one article
- Article list used for per-article summaries only when enabled

4) Summarization + classification
- Per document:
  - Short summary (1â€“2 sentences)
  - Long summary (5â€“7 sentences)
  - Category (from your list; final selection is deterministic rule-based on extracted text)
  - Tags
- Per PDF article:
  - Article summary (4â€“5 sentences, content-first)
  - LLM is used here (summaries, category, tags, article summaries)
- If chat returns `content_filter`, DocAtlas falls back to local extractive summarization
  and continues; document ReviewFlag includes `summary_fallback_content_filter`.

5) Duplicate detection
- Exact duplicates: SHA-256 hash
- Near-duplicates (paraphrase candidates): embeddings + cosine similarity
  - Evaluated within the same category only
  - Balanced rules: strong-score edge or mutual-top1 edge
  - Full text by default (can be changed to summary or disabled)
  - Embeddings API is used here (only if embeddings are enabled)
- Exact + near edges are merged into unified review groups (`ReviewGroupID`)

6) Outputs
- *_docatlas_summaries.xlsx
  - Documents sheet (compact per-document review fields)
  - Duplicates sheet (unified exact+near review members sorted for triage)
  - Articles sheet is conditional:
    - written only when article generation is enabled
    - in append mode, existing `Articles` sheet is preserved unchanged when article generation is disabled
- *_docatlas_import.xlsx (standalone import sheet: Id, Path, Title, Content, Summary, Tags, Attachments, AutoPublish, ArticleType)
- *_docatlas_full_text.xlsx (disabled by default; opt-in via --include-full-text-output)
- summary_report.txt (counts, extraction status, token estimates, no-text list, file type breakdown, category %s, OCR usage, duplicate group stats, doc length stats, per-file errors)
- duplicate_groups_overview.xlsx in each <category>_Duplicate folder (manual cluster assignment/review)
- Optional: resume.json + last_run_stats.json

7) File organization
- Move files into category folders
- Unified review clusters go to <category>_Duplicate/<ReviewGroupID> (exact + near together for review)
- Low/no-text files go to Unreadable
- Files originating from archives are moved from the staging copy; source paths in outputs stay readable as `archive.zip!/inner/file.ext`

Potential cost/time points
==========================
- LLM calls (summaries, categories, tags, article summaries) are the main cost driver.
- Embeddings calls (duplicate detection) add cost, especially with full text.
- OCR adds time (CPU-heavy) but no API cost.
- PDFs with many pages increase OCR time and token usage.
- Large documents increase summarization time and LLM tokens.
- Optional API delay (DOCATLAS_API_DELAY, seconds) can reduce transient socket errors on Windows.
